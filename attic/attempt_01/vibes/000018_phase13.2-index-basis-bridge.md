## Concepts and vocabulary

### Expansion of a rank-1 tensor in a basis

Given a vector `a` and a coordinate system `cs` with covariant basis `{e_0,
e_1, e_2}` and contravariant cobasis `{e^0, e^1, e^2}`, the **covariant
expansion** is:

```
a  =  ÎŁ_i  a^i Â· e_i          (components upper, basis lower)
```

and the **contravariant expansion** is:

```
a  =  ÎŁ_j  a_j Â· e^j          (components lower, cobasis upper)
```

For a general rank-2 tensor:

```
A  =  ÎŁ_{i,j}  A^{ij} e_i âŠ— e_j
```

In tender's expression tree, each term is `Product(component_scalar, basis_vec)`
(or `TensorProduct` for rank > 1).  The components `a^i` become fresh scalar
`NamedTensor` nodes, named e.g. `"a^1"`, `"a^2"`, `"a^3"`.

### World Cartesian System (WCS) shortcut

WCS is orthonormal: `e^i = e_i`, so covariant and contravariant expansions
coincide.  All derivations in the definitional library will use WCS; the
resulting identities hold in any basis by the coordinate-free form.

### Kronecker delta

`IdentityTensor` IS the Kronecker delta in direct notation.  In component form,
`Î´_i^j` is its slot-annotated form.  The evaluation rule for the WCS basis dot:

```
e_i Â· e^j  =  Î´_i^j  =  { 1  if i == j
                          { 0  otherwise
```

For the purpose of derivations, after expansion in WCS the Kronecker delta
appears as a concrete rational (1 or 0), not as a symbolic tensor â€” so the
contraction step is just arithmetic evaluation.

### Kronecker contraction rule

In an explicit sum `ÎŁ_j (... Î´_i^j ...)`, the Î´ collapses the sum:

```
ÎŁ_j  a_j Î´_i^j  =  a_i
```

In tender's explicit-expansion model, this is even simpler: the three-term sum
has two zero terms and one term with coefficient 1, so standard `collect` /
`simplify` reduces it to a single term.

### 2. `expand_in_basis` step

```python
expand_in_basis_step(tensor, cs, covariant=True) -> DerivationStep
```

Given a concrete `NamedTensor` node `tensor` of rank `r` and a `CoordSystem`
`cs`, produces a step that replaces `tensor` with its explicit component
expansion.

For rank 1:
```
a  â†’  Product(NamedTensor("a^0", 0), cs.basis(0))
    + Product(NamedTensor("a^1", 0), cs.basis(1))
    + Product(NamedTensor("a^2", 0), cs.basis(2))
```

Component naming convention: `"{symbol}^{i}"` for upper indices,
`"{symbol}_{i}"` for lower indices.  Component tensors are fresh `NamedTensor`
nodes registered in the global `ResourceList`.

For rank 2 (double expansion), two steps are applied sequentially.

The step is purely a substitution (`replace_in_tree`).

### 3. `simplify_basis_dot` step

```python
simplify_basis_dot_step(cs) -> DerivationStep
```

Walks the expression tree looking for `Contract(basis_i, cobasis_j)` nodes
where `basis_i` and `cobasis_j` belong to `cs`.  By definition of the
cobasis, `e_i Â· e^j = Î´_i^j`, so each such node is replaced with a
`RationalConst`:

- `1`  if `i == j`
- `0`  if `i != j`

Matching is done by pointer comparison against `cs.basis(i)` and
`cs.cobasis(j)`.  For orthonormal systems (e.g. WCS) where cobasis = basis,
`Contract(basis_i, basis_j)` is handled identically.  The more general
`e_i Â· e_j = g_ij` case (both indices covariant) is deferred.

This step does not use pattern matching â€” it compares expression pointers
directly against the `CoordSystem`'s basis and cobasis arrays.

### 4. `collect_zero_terms` step (or extend existing `collect`)

After `simplify_basis_dot`, many terms in a sum become `Scale(0, ...)`.
A `collect_zero_terms_step()` removes them and, if only one non-zero term
remains, unwraps the sum.

This may already be handled by existing simplification if `make_scale(rl, 0,
e)` already folds to zero â€” check during implementation.

**Important limitation**: this step is only sufficient when the result is a
pure scalar â€” i.e. all surviving terms are rational numbers that add up to a
single value.  For tensor-valued results, surviving terms still carry basis
vectors and free indices; they cannot be collected into a scalar and must
instead be reassembled (see deliverable 6 below).

### 5a. When `collect_zero_terms` is not enough

Consider `ÎŁ_j Î´_i^j Î´_j^k` (contraction of two Kronecker deltas).  After
expansion over j and evaluation of each Î´, the 3 non-zero terms are:

```
j=1: {1 if i=1} âŠ— {1 if k=1}
j=2: {1 if i=2} âŠ— {1 if k=2}
j=3: {1 if i=3} âŠ— {1 if k=3}
```

Dropping the 6 zero terms leaves 3 terms, each of rank 2, indexed by (i, k).
The result is `Î´_i^k` â€” but a scalar `collect` step cannot see this; the sum
must be recognised as the component expansion of `IdentityTensor`.

### 6. `reassemble_from_components` step

```python
reassemble_step(cs) -> DerivationStep
```

The inverse of `expand_in_basis_step`.  Walks the expression tree looking for
sums whose terms are all of the form `Product(scalar_expr, basis_vec)` (rank 1)
or `Product(scalar_expr, TensorProduct(basis_i, basis_j))` (rank 2), where all
basis vectors belong to `cs`.

When the sum covers all `dim(cs)` basis directions (i.e. one term per basis
vector with no repetition), the step replaces the whole sum with a fresh
`NamedTensor` whose components match the scalar coefficients â€” or, when those
coefficients are recognised as the components of a known tensor (e.g.
`IdentityTensor`), replaces with that tensor directly.

**Recognition rules for known tensors:**
- `ÎŁ_i 1 Â· (e_i âŠ— e^i)` â†’ `IdentityTensor`
- `ÎŁ_i c Â· e_i` where c is independent of i â†’ `Scale(c * dim, unit_vector)` (degenerate case)
- General case â†’ new `NamedTensor` with recorded component expressions

The step is applied after `collect_zero_terms_step` and operates on whatever
structured sum remains.

**Scope for Phase 13.2**: implement only the rank-1 and rank-2 cases; higher
ranks deferred.

### 8. Scalar commutativity rule

In expressions like `a^i * b_j * Î´_ij`, the component scalars need to commute
past each other.  `Product(s, t)` where both have rank 0 should be
symmetrisable.  This is either:

- A `commute_scalars_step()` derivation step, or
- An identity `s * t = t * s` (for rank-0 operands) added to the definition library.

Decision deferred to implementation; the simpler of the two should be used.

### 9. Basis-independence check (optional, Phase 13.2b)

After a derivation in WCS produces a result, verify that the result is
expressed purely in terms of the original named tensors (no remaining basis
vectors).  If so, the result holds coordinate-free.  This is a sanity check,
not a proof â€” the full proof that WCS is representative requires the tensor
transformation law, which is out of scope.

## Example derivation: `aÂ·b = bÂ·a`

```python
from tender import tensor, dot, wcs
from tender.lib.steps import (
    expand_in_basis_step,
    simplify_basis_dot_step,
    collect_zero_terms_step,
    expand_step,
)

a, b = tensor("a", 1), tensor("b", 1)
cs = wcs()

steps = [
    expand_in_basis_step(a, cs, covariant=True),   # a â†’ ÎŁ a^i e_i
    expand_in_basis_step(b, cs, covariant=False),  # b â†’ ÎŁ b_j e^j
    expand_step(),                                  # distribute dot over sums
    simplify_basis_dot_step(cs),                   # e_iÂ·e^j â†’ Î´_ij (0 or 1)
    collect_zero_terms_step(),                      # drop zero terms
    # result: a^0*b_0 + a^1*b_1 + a^2*b_2
    # symmetry: same expression with aâ†”b labels renamed
]
```

The final expression is manifestly symmetric in the components of `a` and `b`,
completing the proof.

## Design decisions

**Basis expansion produces plain `Sum`, not `ExplicitSum`.**  
Immediate 3-term expansion is simpler and sufficient for 3D.  `ExplicitSum`
deferred until an N-dimensional use case appears.

**Component naming: 1-based.**  
Components are named `"a^1"`, `"a^2"`, `"a^3"` (upper) and `"a_1"`, `"a_2"`,
`"a_3"` (lower) to match standard tensor index notation.

**`simplify_basis_dot` uses cobasis, not metric.**  
For a general `CoordSystem`, `e_i Â· e^j = Î´_i^j` by definition of the
cobasis â€” no need to go through the metric tensor.  The cobasis is already
stored in `cs.cobasis(i)`, so the step compares expression pointers against
`cs.basis(i)` and `cs.cobasis(j)` directly.  The metric path (`e_i Â· e_j =
g_ij`) is a more complex generalisation needed only if both indices are
covariant; defer that to a later phase.

**`reassemble_step` uses pointer comparison.**  
After `simplify_basis_dot`, surviving coefficients are concrete rational
expressions and basis vectors are exactly the pointers stored in the
`CoordSystem`.  The step checks whether the term set forms the complete
`{(e_1, e^1), (e_2, e^2), (e_3, e^3)}` pattern with unit coefficients and
emits `IdentityTensor` if so; otherwise it introduces a fresh `NamedTensor`.
This is adopted as the starting point; revisit if pointer-based matching turns
out to be insufficient in practice.

**Unknown reassembly result â†’ fresh `NamedTensor` with component metadata.**  
The new node is opaque but carries its component expressions as metadata (side
table or derivation annotation).  This metadata can later be used to
re-expand the tensor into its components, making the round-trip lossless.

**Scalar commutativity is an axiom in the definition library.**  
`make_product` almost certainly does not normalise order for rank-0 operands
(verify during implementation).  Rather than a one-off step, add
`scalar_comm: s * t = t * s` (for rank-0 pattern variables `s`, `t`) to
`tender.lib.identities.definitions`.  This makes the rule available to
`search_apply` across all derivations, not just the basis-expansion pipeline.
