# Phase 13.6 — Component-form bridge: indexed notation and invariant reconstruction

## Motivation

Phase 13.2 built a pipeline that goes from an invariant expression `a·b` down
to an explicit component sum `a^1 b_1 + a^2 b_2 + a^3 b_3`.  Phase 13.6
completes the bridge in both directions and adds a compact intermediate
notation.

Three related but distinct capabilities are needed:

1. **Indexed-sum notation** — collapse the explicit sum into the compact
   Einstein form `a^i b_i` (display only; does not reconstruct the invariant).
2. **Invariant reconstruction via atomic steps** — a set of individual
   derivation steps that reverse the component expansion one operation at a
   time, ending at the original invariant expression.
3. **Equality via common component form** — establish that two invariant
   expressions are equal by showing they reduce to the same component form
   under a variable substitution, without reconstructing either from
   components.

---

## Part 1 — Indexed-sum notation (`collect_repeated_sum_step`)

### Requirement

Add a derivation step `collect_repeated_sum_step(cs, index=None)` that:

1. Matches a `Sum` of exactly `cs.dim()` terms where every term is a
   `Product(NamedTensor, NamedTensor)` or `TensorProduct(NamedTensor,
   NamedTensor)` and the two operand symbols share a common structure
   differing only by a running 1-based integer suffix
   (e.g. `a^1 b_1`, `a^2 b_2`, `a^3 b_3`).
2. Emits a new **display-only** AST node `IndexedSum` that renders as
   `a^{i} b_{i}` in LaTeX.
3. Is rank-agnostic: `IndexedSum.rank()` equals the rank of the individual
   terms. (`Product` terms give rank 0; `TensorProduct` terms give the sum of
   the operand ranks.)
4. Passes through non-matching expressions unchanged.

### Index name safety

The chosen letter must not coincide with any index letter already present in
the expression (`ExplicitSum`, `NoSum`, or any future bound-index node).

- **Auto-generation**: scan the expression and pick the first unused letter
  from `i, j, k, l, m, n, p, q, r, s`.
- **User override**: `collect_repeated_sum_step(cs, index="k")` — raises an
  error if the supplied letter is already in use.

### `IndexedSum` AST node

```cpp
class IndexedSum : public Expr {
public:
    // lhs_latex / rhs_latex are the base symbols (without suffix).
    // separator_lhs / separator_rhs are "^" or "_".
    IndexedSum(std::string lhs_latex, std::string sep_lhs,
               std::string rhs_latex, std::string sep_rhs,
               std::string index_letter, int rank);
    auto rank()   const noexcept -> int    override;
    auto latex()  const          -> std::string override;
    auto python() const          -> std::string override;
};
```

`latex()` produces e.g. `a^{i} b_{i}` from the stored parts.  Storing parts
separately (rather than a pre-rendered string) keeps the node inspectable.
`IndexedSum` is terminal: no rewrite rules consume it.

### Note: abstract basis mode for `expand_in_basis_step`

The current `expand_in_basis_step` expands e.g. `a·b` into 9 concrete terms
(all cross-products `a^i e_i · b_j e^j` for i,j ∈ {1,2,3}) before
`simplify_basis_dot_step` reduces it to 3 non-zero terms.

A future **abstract mode** should instead expand directly into a single
`IndexedSum` node:

    a·b  →  Contract(Σ_i (a^i e_i), Σ_j (b_j e^j))

where `Σ_i` is an `ExplicitSum` (or the new `IndexedSum`) over an abstract
index symbol `i`, and `a^i`, `b_j` are NamedTensors with symbolic index slots.
`simplify_basis_dot_step` would then recognise `ExplicitSum` nodes and collapse
`e_i · e^j = δ_i^j` with the implicit Kronecker delta.

The result would be the same `a^i b_i` but with 4 steps total and just 1 term
at the final step — no 9-term explosion.  This is purely a performance and
readability improvement; the concrete-index path remains valid.

## Part 2 — Invariant reconstruction via atomic steps

These steps reverse the Phase 13.2 pipeline one operation at a time.  Each
step is narrow and independently applicable in many situations.

### 2a — `introduce_kronecker_step(cs, index1, index2)`

Replaces an `IndexedSum` (or any expression of the form `Σ_i f(i)`) by
inserting an explicit Kronecker delta that splits the single dummy index into
two independent indices:

    a^i b_i  →  a^i b_j δ_i^j

The two index letters are supplied by the caller (no auto-generation needed
here — the caller is explicitly constructing the split).  The step inserts
`Contract(basis(i), cobasis(j))` in place of the implicit contraction.

This is the only "creative" step — it introduces structure not present in the
current expression, so it must be user-initiated.

### 2b — `delta_to_basis_dot_step(cs)`

Replaces each `Contract(basis(i), cobasis(j))` with the explicit
basis-dot expression, the reverse of `simplify_basis_dot_step`.

    δ_i^j  →  e_i · e^j

This is an identity driven by the definition `e_i · e^j = δ_i^j`.

### 2c — `factor_step()`

Reverse of `expand_step`: collects distributed products back under a single
outer operation.  Applied to

    a^i (e_i · (b_j e^j))

to yield

    (a^i e_i) · (b_j e^j)

This step is algebraically general (not specific to component reconstruction)
and has broad applicability.

### 2d — `reassemble_vector_step(cs)`

The most general reassembly step.  Recognises a sub-expression of the form

    a^1 e_1 + a^2 e_2 + a^3 e_3

and replaces it with the named tensor `a`, inferring the name from the
component naming convention (`{sym}^{k}` → tensor `{sym}`).

    Σ_k (a^k e_k)  →  a

This is the direct inverse of `expand_in_basis_step`.

**Common reassembly shorthand steps** (built on top of `reassemble_vector_step`):

- `reassemble_dot_step(cs)` — recognises `Contract(Σ_i a^i e_i, Σ_j b_j e^j)`
  → `Contract(a, b)`.  Equivalent to applying `reassemble_vector_step` twice
  inside a contraction.
- Further shorthands (cross product, tensor product, trace) deferred to later
  phases as needed.

## Part 3 — Equality via common component form

A common proof strategy avoids reconstructing either side in full.  Given two
invariant expressions A and C, both are expanded to component form; if the
component forms are equal up to a variable substitution (relabelling), A = C
follows immediately.

Example (dot-product commutativity):

    A = a·b  →  a^i b_i
    C = b·a  →  b^i a_i  =  a^i b_i   (swap names a↔b, then scalar_comm)

This is "A ↔ component form" and "C ↔ component form", then observe equality
— shorter than the full chain A ↔ component form ↔ C.

### Design

No new AST node is needed.  The capability is provided by:

1. Expanding both sides to component form using the existing
   `expand_in_basis_step` pipeline.
2. Applying `scalar_comm` (or other commutation identities) to one side to
   match the other.
3. Inspecting the two histories: if the terminal expressions are structurally
   equal, the original expressions are equal.

A convenience function `prove_equal_by_components(lhs, rhs, cs, rules=None)`
can automate steps 1–3 and return a proof certificate (pair of histories).
This is a Python-level utility, not a new C++ node.

---

## Index name safety (applies to all steps that introduce an index letter)

Scan the whole expression for bound index letters before choosing a fresh one.
Preference list: `i, j, k, l, m, n, p, q, r, s`.  User-supplied letters are
validated against the same set.

## Deliverables

1. `IndexedSum` AST node + `collect_repeated_sum_step` (Part 1).
2. `introduce_kronecker_step`, `delta_to_basis_dot_step` (Part 2a–b).
3. `factor_step` (Part 2c).
4. `reassemble_vector_step` + `reassemble_dot_step` (Part 2d).
5. `prove_equal_by_components` convenience function (Part 3).
6. Tests for each deliverable.
7. Updated `dot_commutativity.py` demonstrating both the indexed-sum notation
   (Part 1 exit) and the full reconstruction chain (Part 2 exit).

## Exit criterion

Two proofs of `a·b = b·a` exist in `dot_commutativity.py`:

- **Short proof** (Part 3): expand both sides to component form, apply
  `scalar_comm`, observe equality.
- **Long proof** (Parts 1 + 2): arrive at `a^{i} b_{i}`, reconstruct
  step-by-step back to `b·a`.
