# Phase 13.5 — Identity derivation and library growth

## Motivation

The current identity library (`tender.lib`) contains a handful of ready-made
results: BAC-CAB, anti-commutativity, and two identity-tensor rules.  These
were asserted without proof.  Phase 13.5 takes a different stance:

> **State only the definitions.  Derive everything else.**

Starting from a minimal set of axiomatic rules, we use the same rewrite
machinery the user has — `search_apply`, `apply_identity_auto`,
`Identity.from_derivation` — to build up the library incrementally.  Every
non-definitional entry carries a derivation trace that can be replayed and
inspected.  This has two benefits:

1. **Ground truth** — the library is internally consistent; no identity can
   contradict another without the derivation failing.
2. **Pedagogical value** — the derivation of BAC-CAB, for instance, becomes
   a worked example that demonstrates the system itself.

---

## What counts as a definition

In the direct (coordinate-free) notation tender uses, a *definition* is a rule
that *introduces* a symbol by relating it to other already-defined symbols.  The
smallest closed set of definitions for 3D vector/tensor algebra is:

The hierarchy is: dot product is primitive; trace of a dyad is defined in terms
of it; linearity then extends trace to all rank-2 tensors.

| Symbol | Definition |
|---|---|
| `a · b` (dot product) | Bilinear + basis evaluation: `e_i · e^j = δ_i^j`; commutativity `a·b = b·a` is *derived*, not assumed |
| `tr` (trace) | **`tr(a ⊗ b) = a · b`** — defines trace for a dyad; linearity (`tr(A + B) = tr(A) + tr(B)`) extends to all rank-2 tensors |
| **I** (identity tensor) | `I · a = a` (left and right); `I : A = tr(A)` is then a *derived* consequence of the trace definition |
| `a × b` (cross product) | `a × b = ε : (a ⊗ b)` — double contraction of rank-3 with rank-2 gives rank-1 |
| **ε** (Levi-Civita) | `eps_antisym` + `eps_norm` — all component values follow; everything else is derived |

In practice the definitional layer for the library is:

```
dot_bilinear:      (a + b) · c  =  a·c + b·c          [and symmetric counterpart]
dot_basis:         e_i · e^j    =  δ_i^j               [evaluated by simplify_basis_dot]
tp_contract_right: (u ⊗ v) · w  =  (v · w) u            [dyad contracts from right]
tp_contract_left:  w · (u ⊗ v)  =  (w · u) v            [dyad contracts from left]
trace_dyad:        tr(a ⊗ b)    =  a · b                [defines trace; linearity extends it]
identity_vec:      I · a         =  a                    [defines I on vectors]
cross_def:         a × b          =  ε : (a ⊗ b)          [rank-3 : rank-2 → rank-1; (a×b)_k = ε_{ijk} a^i b^j]
eps_antisym:       ε_ijk         = -ε_jik = -ε_ikj       [anti-symmetry under any swap; index-based]
eps_norm:          ε₁₂₃          =  1                    [normalization; index-based]
scalar_comm:       s t            =  t s                  [commutativity of rank-0 products; axiom]
```

All ε component values follow from `eps_antisym` + `eps_norm` alone.  All
higher-level identities — `eps_contract`, `eps_sq`, `ε·a = -(I×a)`, etc. —
are derived results (see roadmap below).

## Derivation roadmap

Identities to derive in order, from the definitional layer:

### Tier 1 — immediate from definitions

| Identity | Derived from |
|---|---|
| `a · b = b · a` | `dot_basis` + `dot_bilinear` + expand in WCS (Phase 13.2) |
| `I = Σ_i e_i ⊗ e^i` | `identity_vec` + `tp_contract_right` + `dot_basis` + `scalar_comm` + basis expansion |
| `I : A = tr(A)` | `I = Σ_i e_i ⊗ e^i` + `trace_dyad` + linearity of trace |
| `ε : (a ⊗ b ⊗ c) = (a×b) ⊗ c` | `cross_def` + contraction associativity; note rank-2 on both sides |
| `a × b = -(b × a)` | `eps_antisym` applied to `cross_def` |
| `a × a = 0` | special case of anti-commutativity |
| `(λa) · b = λ(a·b)` | `dot_bilinear` + `scalar_comm` |
| `a · (b + c) = a·b + a·c` | `dot_bilinear` |
| `ε·a = -(I × a)` | `cross_def` + `identity_vec` + `eps_antisym` + Phase 13.2 index expansion (equality of components requires enumeration) |
| `eps_contract`: `ε_ijk ε_ilm = δ_jl δ_km - δ_jm δ_kl` | Phase 13.2 index expansion: enumerate Σ_i ε_ijk ε_ilm over all 81 (j,k,l,m) cases using `eps_antisym` + `eps_norm`; elegant alternative via matrix determinant identity deferred until matrices are available |
| `eps_sq`: `ε : ε = 6` | `eps_contract` + `dot_basis` (sum δ_jj δ_kk - δ_jk δ_kj over j,k) |

### Tier 2 — one or two steps from Tier 1

| Identity | Path |
|---|---|
| BAC-CAB: `a×(b×c) = b(a·c) - c(a·b)` | `eps_contract` + `cross_def` → expand → collect |
| Double-cross: `(a×b)×c = b(a·c) - a(b·c)` | anti-commutativity → BAC-CAB |
| Scalar triple product cyclic: `a·(b×c) = b·(c×a) = c·(a×b)` | `cross_def` + `eps_antisym` (cyclic permutation) |
| Jacobi: `a×(b×c) + b×(c×a) + c×(a×b) = 0` | three BAC-CAB applications + collect |
| Lagrange: `(a×b)·(c×d) = (a·c)(b·d) - (a·d)(b·c)` | `eps_contract` |
| `(ε·a):(ε·b) = 2(a·b)` | `eps_contract` + `dot_basis` |

### Tier 3 — standard calculus / mechanics identities

| Identity | Notes |
|---|---|
| `∇(f g) = f ∇g + g ∇f` | product rule for gradient |
| `∇·(f **v**) = ∇f · **v** + f ∇·**v**` | product rule for divergence |
| `∇×(f **v**) = ∇f × **v** + f ∇×**v**` | product rule for curl |
| `∇·(∇×**v**) = 0` | curl is divergence-free |
| `∇×(∇f) = 0` | gradient is curl-free |
| `∇×(∇×**v**) = ∇(∇·**v**) - Δ**v**` | vector Laplacian identity |

Tier 3 requires the differential operator layer to be connected; derivation
proofs will reference coordinate expressions.

## Methodology: derive-then-register

The standard workflow for a new identity:

```python
from tender import tensor, cross, search_apply, Identity, Derivation, State
from tender.lib.identities.epsilon import bac_cab, anti_commutativity

# 1. State the starting expression.
a, b, c = tensor("a", 1), tensor("b", 1), tensor("c", 1)
expr = cross(cross(a, b), c)

# 2. Find the derivation automatically.
steps = search_apply(bac_cab, expr)
history = Derivation(steps).apply(State(expr))

# 3. Register the result as a named identity.
double_cross = Identity.from_derivation("double-cross", history)

# 4. Add it to a library module.
ALL = [bac_cab, anti_commutativity, double_cross]
```

`Identity.from_derivation` captures the LHS (initial expression, with concrete
vectors replaced by pattern variables) and the RHS (final expression).  The
derivation history is currently *not* stored inside the `Identity` object; a
future `Theorem` type (Phase 14+) will carry it.

### Pattern-variable promotion

`Identity.from_derivation` takes concrete tensors and, if told which to treat
as pattern variables, rewrites the LHS and RHS accordingly.  The API sketch:

```python
a_var = make_pattern_var("a"); a_var.constrain_rank(1)
b_var = make_pattern_var("b"); b_var.constrain_rank(1)
c_var = make_pattern_var("c"); c_var.constrain_rank(1)

double_cross = Identity.from_derivation(
    "double-cross", history,
    substitute={a: a_var, b: b_var, c: c_var},
)
```

This is a `substitute_step`-based transformation applied to both endpoints
of the history.  The exact API is to be designed in Phase 13.5.

## User extension workflow

Two contexts to support cleanly:

### A. Extending `tender.lib`

1. Add a Python file under `python/tender/lib/identities/` or
   `python/tender/lib/objects/`.
2. State definitional identities at module level.
3. Derive compound identities using `search_apply` + `Identity.from_derivation`
   at module level (runs once on import).
4. Expose `ALL` list.
5. Add to `python/tender/lib/identities/__init__.py`.

The module is pure Python; no C++ changes required.

### B. Custom user module

```python
# myproject/mechanics_identities.py

from tender import Identity, make_pattern_var, cross, dot, tp
from tender.lib.identities.epsilon import bac_cab, anti_commutativity

# Define project-specific tensors
sigma = make_pattern_var("sigma"); sigma.constrain_rank(2); sigma.constrain_symmetric()
eps_s  = make_pattern_var("eps");  eps_s.constrain_rank(2);  eps_s.constrain_symmetric()

# Constitutive law (example — user asserts, not derived)
hookes_law = Identity(
    "Hooke's law",
    lhs=sigma,
    rhs=...,
)

ALL = [hookes_law]
```

The user passes `rules=ALL` (or `rules=tender.lib.ALL + ALL`) to `search_apply`.

### Desired ergonomics improvements

- A `derive_identity(name, start_expr, target_identity, rules=None)` convenience
  function that wraps `search_apply` + `Identity.from_derivation` in one call.
- A `register(identity, module_ALL_list)` helper that appends and deduplicates.
- Documentation template (Markdown + docstring) for new library modules.

## Design decisions

**`Identity.from_derivation` does not store the proof trace.**  
An `Identity` remains `(name, lhs, rhs)`.  A future `Theorem` type (Phase 14+)
will carry the derivation history.  Keep the two separate.

**Pattern-variable promotion: infer from the concrete tensors.**  
API: `Identity.from_derivation("double-cross", history, pattern_vars=[a, b, c])`.  
`from_derivation` reads the name and rank of each concrete tensor, creates
`PatternVar` nodes internally, substitutes them into both endpoints of the
history, and returns the resulting `Identity`.  No pattern variables leak into
the caller's namespace.  Revisit if the inferred constraints prove insufficient
in practice.

The `substitute` dict approach (caller creates vars explicitly) is available as
an escape hatch for cases where inferred constraints are wrong, but should not
be the default path.

**Minimal Phase 13.5 scope: Tier 1 + Tier 2.**  
Ship all pure vector identities (dot, cross, ε, scalar triple product, BAC-CAB,
double-cross, Jacobi, Lagrange).  Tier 3 (differential operators) requires the
differential operator layer; defer to Phase 13.5b or Phase 15.

**Index-based derivations require Phase 13.2 tooling.**  
Phase 13.2 (index/basis bridge) delivers `expand_in_basis_step`,
`simplify_basis_dot_step`, and explicit index enumeration.  These are
prerequisites for Phase 13.5; `eps_contract` is derived by brute-force
enumeration using that tooling, not stated as an axiom.
