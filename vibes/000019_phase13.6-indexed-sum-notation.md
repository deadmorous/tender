# Phase 13.6 — Indexed-sum notation (collect repeated component sums)

## Motivation

After a WCS component expansion the final expression is an explicit sum such as

    a^1 b_1 + a^2 b_2 + a^3 b_3

This is mathematically correct but visually verbose.  The compact form

    a^i b_i   (Einstein summation over i = 1 … dim)

conveys the same information and is the standard notation in tensor calculus.

A `collect_repeated_sum_step` derivation step that recognises and collapses such
sums is the missing link between Phase 13.2's component-expansion pipeline and
readable mathematical output.

## Requirement

Add a derivation step `collect_repeated_sum_step(cs)` that:

1. Matches a `Sum` of exactly `cs.dim()` terms where every term is a `Product`
   of two rank-0 `NamedTensor` nodes whose symbols share a common structure
   differing only by a running 1-based index suffix
   (e.g. `a^1 b_1`, `a^2 b_2`, `a^3 b_3`).
2. Emits a new AST node `IndexedSum(body, index_letter)` that:
   - renders in LaTeX as `a^{i} b_{i}` (substituting the concrete suffix with
     the free letter `i`);
   - reports `rank() == 0` (it is a scalar);
   - has a meaningful `python()` representation for inspection.
3. Is applied as an ordinary `DerivationStep` so it appears in derivation
   history and LaTeX output, just like any other step.
4. Passes through expressions that do not match the pattern unchanged.

## Index name safety

The index letter chosen for `a^{i} b_{i}` must not coincide with any index
letter already present in the expression — otherwise the output is ambiguous
or outright wrong (e.g. a nested sum that already uses `i`).

**Resolution strategy — auto-generation with optional override:**

- `collect_repeated_sum_step` scans the expression for all index letters
  already in use (visiting every `ExplicitSum`, `NoSum`, and any other node
  that carries a bound index).
- It then picks the first unused letter from a fixed preference list:
  `i, j, k, l, m, n, p, q, r, s`.
- The user may supply an explicit letter via an optional parameter
  `index="k"`.  If the supplied letter is already in use the step raises an
  error rather than silently producing incorrect output.

This keeps the common case effortless while giving the user full control when
needed.

## Scope and non-goals

- The step handles sums of any rank: `Product` (rank-0 × rank-0) and
  `TensorProduct` (any rank combination) are both matched by the same
  pattern — the running integer suffix logic is rank-agnostic.
  `IndexedSum.rank()` returns the rank of the individual terms.
- `IndexedSum` is a **display node** only.  No rewrite rules operate on it;
  it is terminal.  Arithmetic with `IndexedSum` nodes is not required.
- Multi-index or nested sums are deferred.

## Design sketch

### New AST node (`expr.hpp` / `expr.cpp`)

```cpp
class IndexedSum : public Expr {
public:
    IndexedSum(std::string body_latex, std::string index_letter);
    auto rank() const noexcept -> int override { return 0; }
    auto latex()  const -> std::string override;
    auto python() const -> std::string override;
private:
    std::string body_latex_;    // pre-rendered template with index substituted
    std::string index_letter_;
};
```

`latex()` returns the `body_latex_` string directly (the step has already
substituted the concrete suffix with `index_letter_`).

Alternatively, store the two operand LaTeX strings and the index separately so
`latex()` produces `a^{i} b_{i}` from parts — cleaner for inspection.

### Pattern matching in `collect_repeated_sum_step`

Signature: `collect_repeated_sum_step(cs, index=None)`

Given a `Sum` with `dim` terms, for each term:
- Check it is a `Product(NamedTensor*, NamedTensor*)` or
  `TensorProduct(NamedTensor*, NamedTensor*)` (both operands must be
  `NamedTensor` nodes; mixed-rank pairs are fine).
- Parse each symbol as `base + separator + suffix` where `separator` is `^` or
  `_` and `suffix` is the 1-based integer string `"1"`, `"2"`, … `"dim"`.
- Verify that the bases are consistent across all terms and the suffixes form a
  complete 1 … dim sequence.

If matching succeeds:
- Determine the index letter: use the caller-supplied `index` if given
  (error if already in use), otherwise pick the first letter from
  `i, j, k, l, m, n, p, q, r, s` not already present in the expression.
- Replace the concrete suffix with the chosen letter in both operand LaTeX
  strings and return an `IndexedSum` node.

### Python binding and `__init__.py`

- Expose `IndexedSum` type and `collect_repeated_sum_step` via `_tender.cpp`.
- Add to `tender/__init__.py` imports and `__all__`.

## Deliverables

1. `IndexedSum` AST node (C++ + Python binding).
2. `collect_repeated_sum_step(cs)` — C++ implementation + Python binding.
3. Unit tests in `tests/basis_test.cpp` (or a new `indexed_sum_test.cpp`).

## Exit criterion

The `dot_commutativity.py` derivation terminates with the step

    [collect_repeated_sum]  a^{i} b_{i}

visible in `show(history)` and in the compiled PDF.
