# Phase 13.6 â€” Indexed-sum notation (collect repeated component sums)

## Motivation

After a WCS component expansion the final expression is an explicit sum such as

    a^1 b_1 + a^2 b_2 + a^3 b_3

This is mathematically correct but visually verbose.  The compact form

    a^i b_i   (Einstein summation over i = 1 â€¦ dim)

conveys the same information and is the standard notation in tensor calculus.

A `collect_repeated_sum_step` derivation step that recognises and collapses such
sums is the missing link between Phase 13.2's component-expansion pipeline and
readable mathematical output.

## Requirement

Add a derivation step `collect_repeated_sum_step(cs)` that:

1. Matches a `Sum` of exactly `cs.dim()` terms where every term is a `Product`
   of two rank-0 `NamedTensor` nodes whose symbols share a common structure
   differing only by a running 1-based index suffix
   (e.g. `a^1 b_1`, `a^2 b_2`, `a^3 b_3`).
2. Emits a new AST node `IndexedSum(body, index_letter)` that:
   - renders in LaTeX as `a^{i} b_{i}` (substituting the concrete suffix with
     the free letter `i`);
   - reports `rank() == 0` (it is a scalar);
   - has a meaningful `python()` representation for inspection.
3. Is applied as an ordinary `DerivationStep` so it appears in derivation
   history and LaTeX output, just like any other step.
4. Passes through expressions that do not match the pattern unchanged.

## Scope and non-goals

- The step only needs to handle sums of **scalar products** (rank-0 Ă— rank-0)
  that differ by a running integer suffix.  Higher-rank analogues (e.g.
  collapsing `ÎŁ_i e_i âŠ— f_i`) are out of scope for this phase.
- `IndexedSum` is a **display node** only.  No rewrite rules operate on it;
  it is terminal.  Arithmetic with `IndexedSum` nodes is not required.
- The `index_letter` is always `i` for now.  Multi-index or nested sums are
  deferred.

## Design sketch

### New AST node (`expr.hpp` / `expr.cpp`)

```cpp
class IndexedSum : public Expr {
public:
    IndexedSum(std::string body_latex, std::string index_letter);
    auto rank() const noexcept -> int override { return 0; }
    auto latex()  const -> std::string override;
    auto python() const -> std::string override;
private:
    std::string body_latex_;    // pre-rendered template with index substituted
    std::string index_letter_;
};
```

`latex()` returns the `body_latex_` string directly (the step has already
substituted the concrete suffix with `index_letter_`).

Alternatively, store the two operand LaTeX strings and the index separately so
`latex()` produces `a^{i} b_{i}` from parts â€” cleaner for inspection.

### Pattern matching in `collect_repeated_sum_step`

Given a `Sum` with `dim` terms, for each term:
- Check it is a `Product(NamedTensor*, NamedTensor*)`.
- Parse each symbol as `base + separator + suffix` where `separator` is `^` or
  `_` and `suffix` is the 1-based integer string `"1"`, `"2"`, â€¦ `"dim"`.
- Verify that the bases are consistent across all terms and the suffixes form a
  complete 1 â€¦ dim sequence.

If matching succeeds, replace the concrete suffix with `index_letter` in both
operand LaTeX strings and return an `IndexedSum` node.

### Python binding and `__init__.py`

- Expose `IndexedSum` type and `collect_repeated_sum_step` via `_tender.cpp`.
- Add to `tender/__init__.py` imports and `__all__`.
